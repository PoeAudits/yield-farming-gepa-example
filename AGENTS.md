## Project-Specific Conventions

### Evaluator Protocol
Evaluators must conform to the signature `(dict[str, str], dict[str, Any]) -> tuple[float, dict[str, Any]]`:
- **Input 1:** candidate â€” a `dict[str, str]` mapping named section keys to their string values.
- **Input 2:** example â€” a `dict[str, Any]` containing the dataset example for this evaluation. **The parameter must be named `example`** because GEPA passes it as a keyword argument (`example=...`).
- **Output:** a `tuple` of `(score: float, asi: dict[str, Any])` where `score` is the scalar fitness value and `asi` is Actionable Side Information.

### Seed Candidate Format
Seed candidates are always `dict[str, str]` with named section keys (e.g., `{"system": "...", "user": "..."}`). Never use positional lists or raw strings.

### Actionable Side Information (ASI)
Evaluators must always return an `asi` dict as the second element of the return tuple, even if empty (`{}`). ASI passes structured feedback (e.g., failure reasons, partial scores, debug info) back to the optimizer for use in generating better candidates.

### Config Presets
Use the built-in presets when configuring optimizer runs:
- `quick_preset` â€” fast iteration, fewer trials, suitable for development.
- `standard_preset` â€” balanced speed and quality, suitable for most use cases.
- `thorough_preset` â€” maximum exploration, more trials, suitable for production runs.

### Interactive Proposer Handoff Protocol
The `InteractiveProposer` replaces GEPA's automated reflection with a human-in-the-loop review cycle. Each iteration pauses for a human to review failures and write a new prompt candidate.

**Iteration directory structure** (under the handoff directory, e.g. `examples/yield_farming_classifier/handoff/`):
```
iteration_001/
  report.md        # failure analysis generated by ReportGenerator
  prompt_used.md   # the prompt candidate evaluated in this iteration
  new_prompt.md    # written by the human reviewer; read by InteractiveProposer on resume
```

**Communication channel:** `new_prompt.md` is the sole handoff file. The human (or a review agent) writes the full new prompt text into this file. On resume, `InteractiveProposer` reads it and returns it as the next candidate for GEPA to evaluate.

**Start/resume lifecycle:**
- **`make optimize-interactive`** (start): Seeds a new session, evaluates the seed prompt, writes `report.md` + `prompt_used.md` to `iteration_001/`, then exits. GEPA saves state to `gepa_state.bin`.
- Human reviews `report.md` and writes a new prompt to `iteration_001/new_prompt.md`.
- **`make optimize-resume`** (resume): Loads GEPA state from `gepa_state.bin`, reads `new_prompt.md`, evaluates the new candidate, and writes the next iteration's report. Repeat until satisfied.
